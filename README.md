# 출석체크 시스템 (React + Google Apps Script API)

회원번호 기반 QR 코드와 직접 검색을 이용한 출석체크 웹앱입니다.
백엔드는 Google Apps Script를 JSON API로만 사용하고, 프런트엔드는 Vite + React로 분리되어 있어 피크타임에도 빠른 응답성을 제공합니다.

## 구조

```
attendance_check/
├── apps-script/
│   └── Code.gs          # JSON API 백엔드 (회원 조회 / 체크인 / 일자·기간 통계)
├── web/                  # Vite + React 프런트엔드
│   ├── package.json
│   ├── vite.config.js
│   ├── index.html
│   ├── .env.example
│   └── src/
│       ├── main.jsx
│       ├── App.jsx       # #/ 체크인, #/dashboard 대시보드 (해시 라우팅)
│       ├── api.js        # fetch 기반 API 클라이언트 + localStorage 캐시
│       ├── Checkin.jsx   # QR 스캔 + 직접 입력 + 대리 체크인
│       ├── Dashboard.jsx # 실시간 현황 + 일자 조회 + 기간 통계
│       └── styles.css
└── README.md
```

## 왜 React + API 구조인가

- `google.script.run` 은 호출 왕복이 500ms~1s 수준이라 체크인 피크타임에 병목이 됩니다.
- React 앱은 **회원 목록을 첫 로딩 시 1회만** 받아 `localStorage`에 캐싱하고, 이후 검색/자동완성은 **100% 클라이언트에서** 처리합니다.
- 서버 호출은 **체크인/통계 조회만** 발생하며, Apps Script 쪽에서도 `CacheService` 로 회원 목록을 캐싱해 스프레드시트 I/O 를 최소화합니다.
- 출입 기록은 날짜별 시트를 만들지 않고 `출입기록_YYYY` 시트에 연 단위로 누적합니다.
- 프런트는 정적 호스팅(Netlify, Vercel, GitHub Pages, 사내 서버 등) 어디든 배포할 수 있습니다.

## 백엔드 설치

1. 회원 정보가 담긴 구글시트를 엽니다 (시트 이름 기본값 `회원목록`).
2. **확장 프로그램 → Apps Script** 를 열고, `apps-script/Code.gs` 내용을 붙여넣습니다.
3. `onOpen` 를 한 번 실행해서 권한을 승인합니다.
4. 구글시트로 돌아가 새로고침하면 **출석체크** 메뉴가 표시됩니다.
   - **신규 회원 QR 생성** : 빈 셀에만 QR 수식 생성 (회원 추가 후 반복 실행 OK)
   - **전체 QR 재생성** : 모든 행을 덮어쓰기
   - **API URL 보기** : 웹앱 배포 URL 표시
   - **회원 캐시 비우기** : 회원 정보 변경 후 즉시 반영하고 싶을 때
5. **배포 → 새 배포 → 유형: 웹 앱**
   - 실행 계정: 나
   - 액세스: 운영에 맞게 (사내만이면 조직, 공개면 `링크가 있는 모든 사용자`)
6. 배포 URL(`.../exec`)을 복사합니다.

### 스프레드시트 요구사항

- 첫 행이 헤더이며 최소 `회원번호`, `성명` 컬럼이 존재해야 합니다.
- 다른 컬럼명/시트명을 사용한다면 `Code.gs` 상단 `CONFIG` 를 수정하세요.

## 프런트 설치 및 실행

```bash
cd web
cp .env.example .env.local
# .env.local 파일을 열어 VITE_API_URL 에 Apps Script 배포 URL 입력
npm install
npm run dev          # 로컬 개발 서버 (http://localhost:5173)
```

프로덕션 빌드:

```bash
npm run build        # dist/ 에 정적 파일 생성
npm run preview      # 빌드 결과를 로컬에서 확인
```

`dist/` 폴더를 Netlify/Vercel/GitHub Pages/사내 Nginx 등에 업로드하면 됩니다.

### 라우팅

- `#/`           : 체크인 화면 (기본, 입구 키오스크/자원봉사자용)
- `#/dashboard`  : 실시간 대시보드
- `#/my-qr`      : 회원 셀프 QR 조회/인쇄/다운로드

해시 라우팅을 사용하므로 정적 호스팅에서 라우터 설정이 필요 없습니다.

### 회원 셀프 QR 배포 안내

회원들에게 `https://your-host/#/my-qr` 링크를 공지(카카오톡, 문자, 이메일, 게시판 등)하면 각자 접속해서:
1. 회원번호 또는 이름을 입력해 조회
2. **인쇄** 버튼으로 QR + 이름 + 회원번호가 담긴 카드를 인쇄
3. **QR 이미지 저장** 버튼으로 PNG 파일 다운로드
4. 또는 스마트폰에서 QR 이미지를 길게 눌러 사진첩에 저장

현재 별도의 본인 인증 단계는 없습니다. (회원번호는 체크인 시 어차피 공개되는 값이므로 큰 보안 리스크는 아닙니다.) 필요하면 전화번호 뒷 4자리 등으로 인증을 추가할 수 있습니다.

## CORS 관련 메모

Apps Script 웹앱은 `POST` + `Content-Type: text/plain` (simple request) 에 대해 자동으로 CORS 헤더를 붙입니다. `api.js` 는 JSON 바디를 보내되 헤더를 `text/plain` 으로 맞춰서 preflight(OPTIONS) 요청을 피합니다.

## 동작 흐름

### 셀프 체크인 (입구 단말기)
1. 태블릿/스마트폰 브라우저로 프런트 URL 접속.
2. 카메라 권한 허용 → 후면 카메라 자동 선택 → QR 스캔 대기.
3. 회원이 QR을 비추면 즉시 토스트 + "○○○ 님 체크인 완료" 표시.

### 자원봉사자 대리 입력
1. 자원봉사자 본인 스마트폰으로 프런트 URL 접속.
2. 하단 **자원봉사자 대리 입력** 체크박스 ON.
3. QR 스캔 또는 **직접 입력** 탭에서 회원번호/성명으로 검색.

### 직접 입력
- 검색은 클라이언트에서 처리되어 지연이 없습니다.
- 회원번호 정확 입력 후 Enter → 즉시 체크인.
- 부분 일치 후보 리스트 탭 → 체크인.

### 대시보드
- 10초마다 `getTodayStats` 를 호출해 오늘 총원/출입/미출입, 최근 20건, 미출입자 리스트(최대 200명)를 갱신합니다.
- 특정일을 입력하면 해당 날짜 참석자 목록을 조회합니다.
- 시작일/종료일을 입력하면 기간 내 회원별 출입 횟수와 출입일 목록을 조회합니다.

## 출입기록 시트 구조

`출입기록_2026` 같은 연도별 시트가 자동 생성되고, 모든 체크인이 한 시트에 누적됩니다.

| 기록ID | 일자 | 회원번호 | 성명 | 체크인시각 | 방식 | 대리입력 | 비고 |
|---|---|---|---|---|---|---|---|
| 2026-04-11_1001 | 2026-04-11 | 1001 | 양태희 | 09:12:34 | qr | N | |
| 2026-04-11_1011 | 2026-04-11 | 1011 | 이근구 | 09:14:02 | search | Y | |

`기록ID` 는 `일자_회원번호` 형식입니다. 같은 날짜에 같은 회원이 다시 체크인하면 중복으로 처리되어 추가 행을 만들지 않습니다.

## 성능 참고치

- 회원 3,000명 기준 `getMembers` 최초 호출 ~1.5s, 이후 캐시 적중 시 ~200ms.
- 체크인 1건 ~300~500ms (스프레드시트 append + 중복 체크 포함).
- 동시 체크인은 `LockService` 로 직렬화되므로 이론적으로 초당 2~3건 처리 가능.
  더 많은 동시성이 필요하면 Firestore/Cloud Run 등으로 전환을 검토하세요.

## 트러블슈팅

- **CORS 에러**: 배포 시 "액세스 권한"을 `링크가 있는 모든 사용자` 로 설정했는지 확인. `.env.local` 의 URL 끝이 `/exec` 인지 확인.
- **회원이 바뀌었는데 반영 안 됨**: 상단 **회원 새로고침** 버튼을 누르거나, 시트 메뉴의 **회원 캐시 비우기** 실행.
- **카메라가 안 켜짐**: HTTPS 환경에서만 카메라 권한이 허용됩니다. 로컬 개발 시 `vite --host` + `https` 플러그인 사용 또는 배포된 HTTPS URL 사용.
